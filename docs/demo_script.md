# Demo Script – KRITIS Alarm & Dependency Lab

## Before the Demo

1. Ensure PostgreSQL is running: `pg_ctl start` or check your service manager.
2. Ensure `.env` is configured with `DATABASE_URL`.
3. Ensure the TEP `.RData` files are present in `Data/`.
4. Start the Flask server: `flask run`

## Demo Flow

### Step 1 – Open Overview Dashboard

Open `http://localhost:5000` in your browser.

> "This is the KRITIS Alarm & Dependency Lab. It processes real industrial process data from the Tennessee Eastman Process benchmark to demonstrate alarm preprocessing, dependency impact analysis, and crisis readiness scoring."

Point to the stat cards:
- Total runs ingested from real `.RData` files
- Total alarms generated from actual sensor deviations
- Latest readiness score

### Step 2 – Show Real TEP Data Background

> "All sensor data in this system comes from the Tennessee Eastman Process benchmark — a widely used open dataset for industrial process monitoring. It is not real maritime infrastructure data, but it provides authentic industrial fault scenarios."

Explain the four dataset roles: fault-free training, fault-free testing, faulty training, faulty testing.

### Step 3 – Open Alarm Dashboard

Navigate to `/alarms`.

> "The alarm page shows alarms generated by comparing each sensor observation against baseline thresholds derived from the fault-free training data."

Filter for **faulty_testing** role. Show the deviation time series chart.

Highlight:
- Different components in different colors (cooling_system, industrial_process, process_control_system)
- Toggle the "Unique only" filter to show how duplicate alarms are removed

> "Alarm flooding is a well-known problem in industrial control. The duplicate filter suppresses repeated alarms from the same sensor within a 3-step time window — keeping only actionable, unique alarm conditions."

### Step 4 – Show Grouped Incidents

Navigate to `/incidents`.

> "These incidents are automatically grouped from the filtered alarms. Each incident represents a cluster of related alarm events in the same time window and component."

Click on a critical incident. Point to:
- `raw_alarm_count` vs `unique_alarm_count`
- `first_time_step` and `last_time_step` — from real TEP data
- Explanation generated from real alarm statistics

### Step 5 – Open Dependency Graph

Navigate to `/dependencies`.

> "This dependency model is a synthetic research configuration. It represents how infrastructure components depend on each other — for example, if the cooling system fails, it cascades into the industrial process, then into port operations, and eventually into municipal crisis coordination."

Click on the **Cooling System** node. Show the downstream impact analysis.

> "The dependency graph is traversed using breadth-first search to find all affected downstream components and assess the overall risk level."

### Step 6 – Run Cascading Simulation

Click **Run Simulation** on a critical incident (from `/incidents`), or navigate to `/simulation`.

> "The cascading simulation estimates how a real incident propagates through the infrastructure dependency chain over time."

Show the impact timeline:
- Minute offsets for each downstream component
- Impact levels: severe → reduced → at_risk → monitoring_required

Point to the BCM recommendation:
> "The recommendation comes from the BCM profile seeded for the most critically affected component — in this case, the recovery action for the cooling system."

Show the continuity risk:
> "The continuity risk is calculated by comparing the estimated interruption duration against the component's Maximum Tolerable Downtime (MTD) and Recovery Time Objective (RTO)."

### Step 7 – Readiness Assessment

Navigate to `/readiness`.

> "The readiness questionnaire asks structured questions about BCM preparedness: dependencies documentation, backup power, communication channels, and exercise history."

Complete the assessment with realistic answers (mixed yes/partial/no).

> "The weighted score aggregates the answers. A score above 75% indicates good readiness, 50–75% partial readiness, below 50% indicates gaps."

### Step 8 – BPMN Process Viewer

Navigate to `/bpmn`.

> "This BPMN 2.0 process model shows the complete crisis response workflow — from alarm receipt through automated validation, incident classification, dependency impact assessment, and crisis team notification. It is a synthetic research illustration of how BPMN can structure crisis response for critical infrastructure operators."

### Closing Statement

> "This prototype uses real open industrial process data for the alarm layer and a synthetic critical-infrastructure model for the crisis-management layer. In a real research project, the dependency model, alarm rules, and BPMN process would be validated through literature review, interviews, workshops, and expert feedback."
